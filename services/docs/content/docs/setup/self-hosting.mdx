---
title: Self Hosting
description: Deployment requirements for a managed or self-hosted Lush install.
---

Configure state, origins, and auth keys explicitly for self-hosted deployments.
Runtime configuration is schema-validated at boot. Missing required values or
invalid numbers, booleans, and origin lists stop the service with a clear config
error before it begins accepting requests.

## Required state

- PostgreSQL for users, organizations, memberships, sessions, audit events, and
  provider configuration.
- Service-owned tables in the shared database by default.
- Optional separate database URLs later if a service needs independent scaling.

## Required auth configuration

- `DATABASE_URL`
- `LUSH_SECRET_KEY`
- `LUSH_APP_ORIGIN`
- `LUSH_AUTH_JWT_PRIVATE_KEY`
- `LUSH_AUTH_JWT_PUBLIC_KEY`

The JWT keys must be an asymmetric keypair. The API signs short-lived access
JWTs; backing services verify them with the public key.

`LUSH_SECRET_KEY` encrypts stored provider credentials and is also HMAC key
material for deterministic refresh-token successors. Rotating it prevents the
immediately previous refresh token from using its grace replay, but current
refresh tokens remain valid.

## Optional runtime configuration

- `LUSH_API_HOST`, default `0.0.0.0`
- `LUSH_API_PORT`, default `7330`
- `LUSH_AGENT_PORT`, default `7331`
- `LUSH_DB_POOL_SIZE`, default `10`
- `LUSH_SESSION_TTL_MS`, default `2592000000`
- `LUSH_REFRESH_TOKEN_GRACE_MS`, default `60000`
- `LUSH_ACCESS_TOKEN_TTL_MS`, default `300000`
- `LUSH_AUTH_JWT_ISSUER`, default `lush-authz`
- `LUSH_AUTH_JWT_AUDIENCE`, default `lush-api`
- `LUSH_AUTH_PUBLIC_SIGNUP`, default `true`
- `LUSH_TRUSTED_PROXIES`, default empty

`LUSH_APP_ORIGIN` is a comma-separated list of browser app origins allowed to
use credentialed auth. Wildcard origins are rejected.

## Auth rate limits and reverse proxies

The API enforces in-process sliding-window limits before password hashing,
session refresh, or email delivery:

| Endpoint | Source-network limit | Target-specific limit |
| --- | ---: | ---: |
| Register | 5/hour | 3/email/hour |
| Login | 300/15 minutes | 5/(email + network)/15 minutes; 40/email/15 minutes |
| Refresh | 2,000/5 minutes | 20/session/5 minutes |
| Request password reset | 60/hour | 3/email/hour |

A successful login clears both target-specific login windows; the broader
source-network tier intentionally counts successful requests as bulk-abuse
protection. Refresh limits use a hash of the stable refresh-token family so
token rotation does not reset the session window. IPv4 sources are keyed by
address and IPv6 sources by `/64`, preventing privacy-address rotation from
bypassing limits or churning limiter state. Rejected requests return `429`
with a `Retry-After` header.

Registration and password-reset target limits deliberately prioritize
email-bombing protection: an attacker can suppress those emails to one address
for the remainder of the window. Login uses a tight `(email, network)` tier
and a much looser email-wide backstop to make targeted login lockout materially
more expensive while still bounding distributed guessing.

Limits apply independently in each API process, so multi-replica deployments
should also enforce a shared limit at the edge. If the runtime cannot determine
the socket peer, the request uses a shared `unknown` source bucket and the API
logs a warning once so the deployment problem is visible to operators.

With no reverse proxy, leave `LUSH_TRUSTED_PROXIES` empty. The API ignores
`X-Forwarded-For`, `X-Real-IP`, and `X-Forwarded-Proto` from direct clients and
keys limits on the socket peer address.

Behind a reverse proxy, set `LUSH_TRUSTED_PROXIES` to the comma-separated IP
addresses or CIDR ranges from which the API actually accepts proxy traffic.
For example:

```dotenv
LUSH_TRUSTED_PROXIES=10.20.0.0/16,2001:db8:1234::/48
```

The API walks `X-Forwarded-For` from the direct peer toward the client and
stops at the first untrusted hop. Every proxy in the chain must be listed, and
the public-facing proxy must replace client-supplied forwarding headers.
Do not trust an internet-wide range such as `0.0.0.0/0`; that lets callers
choose the address used for throttling.

## Email verification

Email verification is required before access is granted. When password
authentication is enabled, startup fails unless `LUSH_EMAIL_DELIVERY` is
`smtp` or `log` and `LUSH_PUBLIC_APP_URL` is set. Production SMTP requires:

- `LUSH_EMAIL_DELIVERY=smtp`
- `LUSH_EMAIL_FROM`
- `LUSH_SMTP_URL` (an `smtp://` or `smtps://` URL)
- `LUSH_PUBLIC_APP_URL`, the browser origin used in verification/reset links

`LUSH_AUTH_SIGNUP_ENABLED` defaults to `true`. `LUSH_AUTH_PUBLIC_SIGNUP` also
defaults to `true`: registration returns the same verification-required
response for new and existing emails, and an existing address receives an
account-exists notification. This prevents unauthenticated account enumeration.
Small private deployments can set `LUSH_AUTH_PUBLIC_SIGNUP=false` to return the
friendlier but enumerable `email_in_use` error. Login always performs a password
KDF even when the email is unknown.

Token TTLs are controlled by `LUSH_EMAIL_VERIFICATION_TTL_MS` (24 hours) and
`LUSH_PASSWORD_RESET_TTL_MS` (one hour). Tokens are random, single-use,
expiring, and stored only as hashes. Successful password resets revoke every
active session.

Verification requires an explicit form submission after the link opens, so
mail security scanners cannot consume the token or verify an account merely by
loading the page. Reset-request responses do not wait for SMTP delivery;
delivery failures are logged without revealing account existence to callers.

Verification and reset tokens are present in link query strings. Lush sends a
`no-referrer` browser policy, but operators should also treat reverse-proxy
access logs as sensitive because they may contain live tokens until expiry.

`LUSH_EMAIL_DELIVERY=log` writes full messages, including bearer links, to the
service log and is for development only. The explicit `bun run
auth:verify-email -- user@example.com` command remains available as an operator
override when email delivery is intentionally unavailable.
